## R1: RPG 式习惯追踪系统

### Phase 1 用户故事

#### US-1.1: 任务创建与管理（CRUD）
- **角色**: 作为已注册用户
- **行为**: 我希望创建、编辑、删除 Daily（日常任务）、Habit（习惯追踪）、Quest（一次性任务）三种类型的任务
- **目的**: 以便分类管理我的日常行为和待办事项
- **来源**: R1（任务类型体系）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户在任务中心页面, When 点击"添加任务"并填写名称、类型（Daily/Habit/Quest）, Then 任务出现在对应分类列表中
  - Given 用户有一个已创建的任务, When 点击编辑按钮修改名称, Then 任务名称更新成功
  - Given 用户有一个已创建的任务, When 点击删除按钮并确认, Then 任务从列表中移除
- **做什么**: 支持 Daily / Habit / Quest 三种类型的增删改查；每个任务包含名称、类型、描述（可选）、精力标签（可选）
- **不做什么**: 不包含 Boss 类型任务（Phase 2）；不支持任务模板批量导入（Phase 3）；不支持任务排序拖拽（Phase 2）

#### US-1.2: 任务完成与 XP 获取
- **角色**: 作为已注册用户
- **行为**: 我希望完成任务后根据难度自动获得 XP，并在经验条上看到即时增长
- **目的**: 以便获得即时正向反馈，维持执行动力
- **来源**: R1（XP 计算引擎）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户有一个未完成的 Daily 任务, When 标记为完成, Then 根据任务难度获得对应 XP，经验条实时增长
  - Given 任务难度为"简单", When 完成时, Then 获得基础 XP（如 10 点）
  - Given 任务难度为"困难", When 完成时, Then 获得更高 XP（如 50 点）
- **做什么**: 任务完成后自动计算 XP（基于难度加权）；经验条动画反馈；记录完成时间戳
- **不做什么**: 不包含连续完成天数的额外加成计算（Phase 2）；不包含 XP 获取历史记录页面（Phase 2）

#### US-1.3: 等级系统
- **角色**: 作为已注册用户
- **行为**: 我希望 XP 累积到阈值后自动升级，并看到升级提示
- **目的**: 以便感知自己的持续成长
- **来源**: R1（等级系统）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户当前等级为 1，经验值距下一级还需 20 点, When 完成一个 30 XP 的任务, Then 等级提升至 2，显示升级提示
  - Given 用户升级, When 升级动画播放完毕, Then 经验条显示新等级下的剩余经验
- **做什么**: 等级阈值表（如 Lv1=100XP, Lv2=200XP 递增）；升级提示弹窗；等级显示在角色卡片上
- **不做什么**: 不包含升级解锁新功能的逻辑（Phase 2）；不包含等级上限机制（Phase 2）

#### US-1.4: 失败惩罚机制
- **角色**: 作为已注册用户
- **行为**: 我希望连续跳过日常任务时系统扣除 HP，模拟惩罚
- **目的**: 以便感受到不执行行为的代价，增强执行动力
- **来源**: R1（失败惩罚机制）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户有一个 Daily 任务, When 连续 2 天未标记完成且未跳过, Then 扣除少量 HP（如 5 点）
  - Given 用户主动跳过 Daily 任务, When 连续跳过 3 天, Then 扣除 HP 并显示警告提示
- **做什么**: 连续未完成检测逻辑；HP 扣除动画；HP 归零时的降级警告
- **不做什么**: 不包含降低等级上限（Phase 2）；不包含 HP 恢复道具机制（Phase 2）

#### US-1.5: 精力标签系统
- **角色**: 作为已注册用户
- **行为**: 我希望为任务标注精力类型（创造性/分析性/机械性/社交性）
- **目的**: 以便系统根据我的精力状态推荐合适的任务
- **来源**: R1（精力标签系统）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户在创建/编辑任务时, When 选择精力标签, Then 任务关联对应的精力类型（创造性/分析性/机械性/社交性）
  - Given 一个任务已标注"创造性"精力标签, When 在任务中心按精力类型筛选, Then 该任务出现在"创造性"分类中
- **做什么**: 4 种精力标签（创造性/分析性/机械性/社交性）；任务创建时可选标注；按精力标签筛选任务
- **不做什么**: 不包含精力值自动计算（Phase 2 R6）；不包含基于精力状态的自动推荐（Phase 2 R6）

#### US-1.6: 习惯打卡与连续追踪
- **角色**: 作为已注册用户
- **行为**: 我希望每日打卡记录习惯完成情况，并看到连续打卡天数
- **目的**: 以便追踪习惯养成的持续性
- **来源**: R1（Habit 习惯追踪）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户有一个 Habit 类型任务, When 今天标记完成, Then 显示连续打卡天数 +1
  - Given 用户昨天未完成 Habit 任务, When 今天打开应用, Then 连续天数重置为 0，显示中断提示
- **做什么**: Habit 任务的每日打卡功能；连续天数计算与显示；中断提示
- **不做什么**: 不包含每周/每月频率设置（Phase 2）；不包含习惯强度热力图（Phase 2 R5）

#### US-1.7: 金币基础获取
- **角色**: 作为已注册用户
- **行为**: 我希望完成任务后获得金币奖励
- **目的**: 以便后续用于购买装备或兑换奖励
- **来源**: R1（金币经济系统）
- **优先级**: Should
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户完成一个任务, When XP 结算时, Then 同时获得对应金币（如简单=5金，困难=20金）
  - Given 用户查看角色卡片, When 金币余额变化, Then 金币数字实时更新
- **做什么**: 任务完成时同步发放金币；金币余额显示在角色卡片
- **不做什么**: 不包含金币消费场景（Phase 2）；不包含商店系统（Phase 2）

#### US-1.8: 基础角色卡片
- **角色**: 作为已注册用户
- **行为**: 我希望在仪表盘看到我的角色信息（头像、等级、HP、XP、金币）
- **目的**: 以便随时了解自己的角色状态
- **来源**: R1（用户模型）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户登录后进入仪表盘, When 页面加载完成, Then 显示角色头像、等级、HP 条、XP 经验条、金币余额
  - Given 用户完成任务获得 XP, When 经验条更新, Then 角色卡片上的数值实时刷新
- **做什么**: 角色卡片组件，展示头像/等级/HP/XP/金币；数据实时更新
- **不做什么**: 不包含装备栏展示（Phase 2）；不包含宠物展示（Phase 3）；不包含角色自定义外观（Phase 3）

## R2: AI 驱动的目标分解引擎

### Phase 2 用户故事

#### US-2.1: 目标输入与 AI 分解
- **角色**: 作为已注册用户
- **行为**: 我希望输入一个中长期目标（如"三个月学会 Python 数据分析"），系统自动拆解为 6-8 个递进式里程碑
- **目的**: 以便将模糊的大目标转化为可执行的步骤
- **来源**: R2（AI 目标分解）
- **优先级**: Must
- **所属阶段**: Phase 2
- **验收标准**:
  - Given 用户在目标分解页面输入"三个月学会 Python 数据分析", When 点击"AI 分解", Then 系统在 3 秒内生成 6-8 个里程碑链（如环境搭建→基础语法→数据处理→可视化）
  - Given AI 生成了里程碑链, When 用户查看每个里程碑, Then 每个里程碑包含名称、描述、预计时长
- **做什么**: 目标输入框；调用 AI 生成 6-8 个里程碑；里程碑链可视化展示
- **不做什么**: 不包含时间梯度子任务分配（US-2.3）；不包含多模型切换（US-2.1 使用默认模型）

#### US-2.2: 统一模型路由层
- **角色**: 作为系统
- **行为**: 系统需要通过统一接口调用不同的 AI 模型（通义千问/智谱GLM/百川/月之暗面）
- **目的**: 以便灵活切换模型供应商，降低绑定风险
- **来源**: R2（AI 集成）+ 分析文档第九节（统一模型路由层）
- **优先级**: Must
- **所属阶段**: Phase 2
- **验收标准**:
  - Given 系统配置了通义千问 API, When 调用目标分解, Then 通过统一路由层成功调用并返回结果
  - Given 切换配置到智谱 GLM, When 再次调用目标分解, Then 使用新模型成功返回结果
- **做什么**: ModelConfig 配置模型；ModelRouter 统一抽象接口；各 provider 独立适配模块
- **不做什么**: 不包含模型性能对比评估（Phase 2 后续）；不包含自动选择最优模型（不在 MVP 范围）

#### US-2.3: 时间梯度分配
- **角色**: 作为已注册用户
- **行为**: 我希望每个里程碑进一步拆分为 10/25/40 分钟梯度的子任务
- **目的**: 以便根据自己的可用时间选择合适的执行单元
- **来源**: R2（时间梯度分配器）
- **优先级**: Must
- **所属阶段**: Phase 2
- **验收标准**:
  - Given AI 已生成里程碑链, When 用户选择某个里程碑, Then 显示该里程碑下的 10/25/40 分钟梯度子任务列表
  - Given 用户选择一个 25 分钟子任务, When 启动专注计时, Then 计时器设置为 25 分钟
- **做什么**: 里程碑下的时间梯度子任务生成；10/25/40 分钟三档选择；与专注计时器联动
- **不做什么**: 不包含自动推荐最佳时长（Phase 2 R6）；不包含历史时长偏好学习（Phase 3）

#### US-2.4: 难度自适应算法
- **角色**: 作为已注册用户
- **行为**: 我希望系统根据我的完成/跳过行为动态调整后续任务的难度
- **目的**: 以便任务难度始终在我的能力范围内，既不太简单也不太难
- **来源**: R2（难度自适应）
- **优先级**: Should
- **所属阶段**: Phase 2
- **验收标准**:
  - Given 用户连续完成 5 个"简单"任务, When AI 生成下一批任务, Then 推荐难度适当提升
  - Given 用户连续跳过 3 个"困难"任务, When AI 生成下一批任务, Then 推荐难度降低
- **做什么**: 基于完成/跳过历史调整难度；调整结果体现在新任务推荐中
- **不做什么**: 不包含多维度能力评估（Phase 3）；不包含社交比较调整（不在 MVP 范围）

#### US-2.5: AI 降级机制
- **角色**: 作为系统
- **行为**: 当 AI 服务不可用时，系统应使用本地模板兜底生成目标分解结果
- **目的**: 以便在网络异常或 AI 服务宕机时核心功能仍可用
- **来源**: R2（降级机制）
- **优先级**: Must
- **所属阶段**: Phase 2
- **验收标准**:
  - Given AI 服务不可用（网络超时或 API 错误）, When 用户请求目标分解, Then 系统在 500ms 内返回本地模板生成的里程碑链
  - Given 本地模板生成结果, When 用户查看, Then 结果标注"模板生成"以区分 AI 结果
- **做什么**: 超时检测（如 3 秒）；本地模板库（至少 3 种常见目标类型）；降级结果标识
- **不做什么**: 不包含离线模式下的完整 AI 功能（不在 MVP 范围）；不包含模板编辑功能（Phase 3）

#### US-2.6: 目标分解 Prompt 模板
- **角色**: 作为系统
- **行为**: 系统需要设计可复用的 Prompt 模板用于目标分解
- **目的**: 以便标准化 AI 输出格式，确保生成质量
- **来源**: R2（Prompt 模板设计）
- **优先级**: Must
- **所属阶段**: Phase 2
- **验收标准**:
  - Given Prompt 模板已配置, When 输入不同类型的目标（学习/健身/职业）, Then AI 输出格式一致且内容合理
  - Given 模板包含约束条件, When AI 返回结果, Then 结果符合 6-8 个里程碑、递进式结构的要求
- **做什么**: 设计通用 Prompt 模板；支持按目标类型微调；结构化输出验证
- **不做什么**: 不包含用户自定义 Prompt（Phase 3）；不包含 Prompt A/B 测试（Phase 3）

## R3: 生活属性与精力可视化仪表盘

### Phase 1 用户故事

#### US-3.1: 三大类属性面板显示
- **角色**: 作为已注册用户
- **行为**: 我希望在仪表盘看到三大类生活属性（生理需求/情感状态/能力属性）的实时状态
- **目的**: 以便全面了解自己的生活状态
- **来源**: R3（三大类属性体系）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户进入属性面板页面, When 页面加载完成, Then 显示三大类属性：生理需求（睡眠、饮食、运动）、情感状态（情绪、压力、社交）、能力属性（技能、知识、创造力）
  - Given 属性面板已显示, When 用户查看某个属性, Then 显示属性名称、当前数值、状态颜色编码
- **做什么**: 三大类属性展示（每类含 3-5 个具体属性）；属性数值显示；分类折叠/展开
- **不做什么**: 不包含 15 种属性的完整实现（Phase 1 先实现核心属性，后续扩展）；不包含属性变化速率模拟（Phase 2）

#### US-3.2: 属性手动更新
- **角色**: 作为已注册用户
- **行为**: 我希望手动输入/更新我的属性数值（如今日饮水量、睡眠时长、情绪评分）
- **目的**: 以便保持属性数据的准确性
- **来源**: R3（属性实时更新）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户在属性面板中, When 点击某个属性的"更新"按钮, Then 弹出输入框允许输入新数值
  - Given 用户输入新数值并确认, When 提交成功, Then 属性面板立即刷新显示新数值和变化趋势
- **做什么**: 手动输入入口；数值校验（合理范围）；更新后即时刷新
- **做什么**: Phase 1 仅支持手动输入；不包含可穿戴设备自动同步（Phase 3）

#### US-3.3: 颜色编码系统
- **角色**: 作为已注册用户
- **行为**: 我希望属性数值用颜色编码直观显示状态（红/黄/绿）
- **目的**: 以便一眼识别哪些属性需要关注
- **来源**: R3（颜色编码）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 某属性数值处于健康范围, When 显示在面板上, Then 使用绿色标识
  - Given 某属性数值处于警告范围, When 显示在面板上, Then 使用黄色标识
  - Given 某属性数值处于危险范围, When 显示在面板上, Then 使用红色标识
- **做什么**: 三级颜色编码（绿/黄/红）；每个属性有独立的阈值配置
- **不做什么**: 不包含自定义颜色方案（Phase 3 多主题）；不包含颜色盲友好模式（Phase 3）

#### US-3.4: 每日启动精力评估
- **角色**: 作为已注册用户
- **行为**: 我希望每天启动应用时看到基于睡眠、起床时长、昨日专注数据的精力评估
- **目的**: 以便了解今天的最佳任务类型
- **来源**: R3（每日启动精力评估流程）
- **优先级**: Must
- **所属阶段**: Phase 1
- **验收标准**:
  - Given 用户每天首次打开应用, When 仪表盘加载, Then 显示精力评估卡片，包含创造力指数、认知负荷、疲劳程度三个维度
