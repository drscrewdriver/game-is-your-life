## 1. 产品概述

### 1.1 愿景与定位

柳比歇夫在《奇特的一生》中展示了"时间统计法"的核心理念：**持续的数据追踪 → 自我认知 → 自律习惯 → 持续改进**。但对大多数人而言，这种方法高度依赖意志力，难以长期坚持。

**产品定位**：数据追踪 × RPG 升级体系 × AI 智能体 × 精力管理，覆盖"即时反馈 → 持续动力 → 自我认知 → 效能优化"的完整闭环。

**核心理念**：将柳比歇夫时间统计法的精神通过游戏化机制落地，让用户在追踪数据的过程中获得即时正反馈，通过 RPG 升级体系维持长期动力，借助 AI 智能体辅助目标分解和精力管理，最终实现自我认知和持续改进。

### 1.2 核心价值主张

| 痛点 | 解决方案 | 价值主张 |
|------|---------|---------|
| 目标难拆解 | AI 驱动的目标分解引擎 | 大目标一键拆解为可执行步骤 |
| 执行无动力 | RPG 升级体系 + 双轨奖励 | 每个行为都有即时游戏化反馈 |
| 成长不可视 | 全维度成长可视化 | 用数据证明自己在成长 |
| 精力管理缺失 | 生活属性 + 精力仪表盘 | 在最佳状态做最合适的事 |
| 拖延与治未病 | 治未病时间管理系统 | 在任务变紧急之前提前干预 |

### 1.3 差异化设计

| # | 差异化点 | 竞品现状 | 本项目策略 |
|---|---------|---------|-----------|
| 1 | AI 深度集成 | 仅 PlotPlayer 有类似能力 | AI 目标分解 + 难度自适应 |
| 2 | 柳比歇夫精神 | 无直接竞品 | 时间统计 → 自我认知 → 持续改进闭环 |
| 3 | 精力管理核心 | **无成熟开源方案** | 精力峰值与任务难度自动匹配 |
| 4 | 治未病时间管理 | **无成熟开源方案** | 艾森豪威尔矩阵游戏化 + 拖延预警 |
| 5 | 多源数据输入 | BeaverHabits 有基础支持 | 可穿戴设备 API 自动更新属性 |
| 6 | 降级机制 | 多数竞品无此设计 | AI 不可用时本地模板兜底 |
| 7 | 自托管优先 | BeaverHabits 有类似架构 | 隐私保护和数据主权 |
| 8 | 拖延症自救模式 | **无成熟开源方案** | 激进干预策略 |

## 2. 目标用户画像

### 2.1 效率追求者 — 小李

| 维度 | 描述 |
|------|------|
| **年龄** | 25-35 岁 |
| **职业** | 互联网从业者 / 自由职业者 |
| **痛点** | 试过多种时间管理方法（番茄钟、GTD），但难以坚持；知道要追踪时间，但缺乏即时反馈 |
| **需求** | 希望有一个系统能自动追踪时间分配，并用游戏化方式激励自己坚持 |
| **使用场景** | 每天早上打开应用查看今日任务 → 完成任务获得 XP → 晚上查看属性面板和成长统计 |
| **技术水平** | 中等，能使用 Docker 部署自托管应用 |

### 2.2 拖延症自救者 — 小王

| 维度 | 描述 |
|------|------|
| **年龄** | 20-30 岁 |
| **职业** | 大学生 / 考研/考公备考者 |
| **痛点** | 明知任务重要但总是拖到最后一刻；尝试过自律但反复失败；需要外部干预 |
| **需求** | 希望有系统能在拖延恶化之前发出预警，并提供强制干预手段 |
| **使用场景** | 启用"拖延症自救模式" → 收到 30 分钟定时检查提醒 → 重要事务预警避免拖延至紧急区间 |
| **技术水平** | 中低，偏好开箱即用的方案 |

### 2.3 RPG 爱好者 — 小张

| 维度 | 描述 |
|------|------|
| **年龄** | 18-28 岁 |
| **职业** | 学生 / 初级职场人 |
| **痛点** | 喜欢 RPG 游戏的升级和收集机制，但市面上的习惯追踪应用缺乏游戏深度 |
| **需求** | 希望把日常生活变成一场 RPG 冒险，完成任务就是打怪升级 |
| **使用场景** | 创建角色 → 完成任务获得 XP 和金币 → 购买装备 → 组队挑战 Boss → 解锁成就 |
| **技术水平** | 中等，熟悉 Habitica 等游戏化应用 |

## 3. 功能规格

### 3.1 Phase 1 功能（核心验证，5-6 周）

> **Phase 1 目标**：最小可行产品，验证"数据追踪 → 即时反馈"闭环

#### F1.1 用户系统 + 基础 RPG 模型

| 项目 | 说明 |
|------|------|
| **描述** | 用户注册/登录，创建角色并初始化 RPG 属性（等级、XP、HP、金币） |
| **用户故事** | US-1.8（基础角色卡片） |
| **交互流程** | 注册 → 创建角色（选择头像）→ 进入仪表盘 → 看到角色卡片（等级1/HP100/XP0/金币0） |
| **验收标准** | 用户可注册登录；角色卡片正确显示所有 RPG 属性；属性值随操作实时更新 |
| **技术约束** | JWT 认证；MongoDB 存储用户文档；Redis 缓存 Session |

#### F1.2 日常任务追踪 + 习惯打卡

| 项目 | 说明 |
|------|------|
| **描述** | 支持 Daily/Habit/Quest 三种任务类型的创建、完成、追踪 |
| **用户故事** | US-1.1（任务CRUD）、US-1.2（XP获取）、US-1.3（等级系统）、US-1.4（失败惩罚）、US-1.5（精力标签）、US-1.6（习惯打卡）、US-1.7（金币获取） |
| **交互流程** | 添加任务（选类型+精力标签）→ 标记完成 → 获得 XP+金币 → 经验条增长 → 升级提示 → 连续跳过扣 HP |
| **验收标准** | 三种类型任务 CRUD 正常；XP 根据难度加权计算；等级正确升级；连续跳过扣 HP；精力标签可选标注；习惯打卡连续天数正确 |
| **技术约束** | XP 公式：基础值 × 难度系数（简单1.0/普通1.5/困难2.0）；等级阈值表可配置 |

#### F1.3 生活属性面板

| 项目 | 说明 |
|------|------|
| **描述** | 三大类属性（生理/情感/能力）追踪面板，含颜色编码和手动更新 |
| **用户故事** | US-3.1（属性面板显示）、US-3.2（手动更新）、US-3.3（颜色编码）、US-3.5（自定义属性）、US-3.6（历史趋势） |
| **交互流程** | 查看属性面板 → 看到三大类属性及颜色编码 → 点击属性手动更新数值 → 查看 7/30 天趋势 |
| **验收标准** | 三大类属性正确显示；颜色编码（绿/黄/红）准确反映状态；手动更新后即时刷新；支持自定义属性；趋势图占位正确 |
| **技术约束** | Phase 1 预设属性约 9-12 个（每类 3-4 个）；颜色阈值可配置 |

#### F1.4 每日启动精力评估

| 项目 | 说明 |
|------|------|
| **描述** | 每天首次启动时基于睡眠、起床时长、昨日专注数据计算精力状态 |
| **用户故事** | US-3.4（每日启动精力评估） |
| **交互流程** | 启动应用 → 读取睡眠+起床+昨日专注数据 → 计算创造力/认知负荷/疲劳三维度 → 显示精力评估卡片 → 输出今日最佳任务类型建议 |
| **验收标准** | 首次启动显示精力评估卡片；三维度指数正确计算；任务类型建议合理 |
| **技术约束** | Phase 1 数据源为手动输入（睡眠时长、起床时间、昨日专注时长）；计算公式待定义（见开放问题 OQ-1） |

#### F1.5 基础前端仪表盘

| 项目 | 说明 |
|------|------|
| **描述** | 整合角色卡片、属性面板、精力评估、任务摘要的首页仪表盘 |
| **用户故事** | US-1.8、US-3.1、US-3.4 |
| **交互流程** | 登录后进入仪表盘 → 顶部角色卡片 → 左侧精力评估+任务摘要 → 右侧属性面板 → 底部快速操作区 |
| **验收标准** | 仪表盘正确整合所有组件；数据实时更新；页面加载 < 3s |
| **技术约束** | React + Tailwind + shadcn/ui；响应式布局（桌面端优先） |

## 4. 非功能需求

### 4.1 性能指标矩阵

| 模块 | 关键指标 | 目标值 | 测量方式 |
|------|---------|--------|---------|
| 仪表盘 | 首屏加载时间（FCP） | < 2s | Lighthouse |
| 属性面板 | 属性手动更新 → UI 刷新 | < 300ms | Performance API |
| AI 引擎 | 目标分解请求 → 首屏展示 | < 3s | Streaming latency |
| AI 引擎 | 降级切换时间 | < 500ms | 超时检测 |
| 热力图 | 365 天渲染 | < 500ms | Performance API |
| API | 数据接口响应时间 | < 200ms | APM |
| 并发 | WebSocket 连接数（Phase 3） | 100 连接/实例 | 压力测试 |
| 存储 | 单用户年数据量 | < 50MB | 数据库统计 |

### 4.2 安全需求

| 需求 | 说明 |
|------|------|
| 用户认证 | JWT Token 认证，支持刷新令牌机制 |
| 密码安全 | bcrypt 哈希存储，最低 8 位，含大小写+数字 |
| API 安全 | Rate Limiting（100 次/分钟/IP）；CORS 白名单 |
| 数据加密 | HTTPS 强制；敏感数据（API Key）加密存储 |
| 数据主权 | 用户可导出全部数据（CSV/JSON）；可删除账户及所有数据 |

### 4.3 可用性需求

| 需求 | 说明 |
|------|------|
| 自托管部署 | Docker Compose 一键部署；完整部署文档 |
| 离线基础功能 | 任务查看和完成在离线状态下可用，联网后同步 |
| 浏览器兼容 | Chrome 90+、Firefox 90+、Safari 15+、Edge 90+ |
| 无障碍 | WCAG 2.1 AA 级别（Phase 3 完善） |

### 4.4 技术约束

> 直接引用 `unified-analysis.md` 第四节技术栈选型

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| 前端框架 | React + Tailwind + shadcn/ui | 组件化 + 主题系统 + 生态成熟 |
| 后端框架 | Python + FastAPI | AI 集成方便、类型安全 |
| 数据库 | MongoDB + Redis | 文档型适合复杂游戏数据结构 |
| AI 集成 | Agently + PydanticAI + CrewAI + LlamaIndex | 评估 + 推理 + 多Agent + RAG |
| 部署方案 | Docker Compose | 自托管 / 隐私保护 |
| 图表可视化 | D3.js / ECharts | 热力图 / 时间线 / 经验曲线 |
| 实时通信 | WebSocket / Socket.IO | 属性实时更新 / 组队协作 |

## 6. 数据模型概要

### 核心实体关系

```
User (用户)
 ├── Character (角色)
 │    ├── level, xp, hp, gold
 │    ├── equipment[] (Phase 2)
 │    └── pet (Phase 3)
 ├── Task (任务)
 │    ├── type: Daily | Habit | Quest | Boss
 │    ├── energy_tag: Creative | Analytical | Mechanical | Social
 │    ├── difficulty, xp_reward, gold_reward
 │    └── status, completed_at, streak_count
 ├── Attribute (属性)
 │    ├── category: Physical | Emotional | Ability
 │    ├── name, value, threshold_min, threshold_max
 │    └── AttributeLog[] (历史记录)
 ├── EnergyAssessment (精力评估)
 │    ├── creativity_index, cognitive_load, fatigue_level
 │    ├── sleep_data, wake_time, yesterday_focus_duration
 │    └── recommended_task_type
 ├── Goal (目标)
 │    ├── title, description, created_at
 │    └── Milestone[] (里程碑)
 │         ├── title, duration_tier (10/25/40)
 │         └── subtask[]
 ├── Achievement (成就)
 │    ├── type, name, description, icon
 │    └── unlocked_at
 └── Party (小队, Phase 3)
      ├── members[]
      └── BossChallenge[] (Phase 3)
```

### 关键数据流

```
任务完成 → XP/金币计算 → 角色属性更新 → 仪表盘刷新
属性更新 → 颜色编码判断 → 面板刷新 → 趋势图更新
精力评估 → 任务类型推荐 → 任务列表筛选
目标输入 → AI 分解 → 里程碑链 → 子任务生成 → 任务列表
```

## 附录 A: 术语表

| 术语 | 定义 |
|------|------|
| Daily | 日常任务，每日重复的待办事项 |

| Quest | 一次性任务，完成即消失的待办事项 |
