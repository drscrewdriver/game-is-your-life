## 一、项目概述

### 核心理念

柳比歇夫在《奇特的一生》中展示的"时间统计法"核心理念是：**持续的数据追踪 → 自我认知 → 自律习惯 → 持续改进**。

但对大多数人而言，这种高度依赖意志力的方法存在五大痛点：

| 痛点 | 说明 | 本项目解决方案 |
|------|------|--------------|
| 目标难拆解 | 中长期目标无法转化为每日可执行步骤 | AI 驱动的目标分解引擎 |
| 执行无动力 | 反馈周期长（以年为单位），缺乏即时正向强化 | RPG 升级体系 + 双轨奖励 |
| 成长不可视 | 感觉自己在成长，但没有量化数据证明 | 全维度成长可视化 |
| 精力管理缺失 | 未区分任务难度与个人精力峰值的匹配 | 生活属性 + 精力仪表盘 |
| 拖延与治未病困境 | 重要事务未提前推进，落入"紧急且重要"区间 | 治未病时间管理系统 |

### 项目定位

> **数据追踪 × RPG 升级体系 × AI 智能体 × 精力管理**，覆盖"即时反馈 → 持续动力 → 自我认知 → 效能优化"的完整闭环。

### 差异化设计方向

| # | 差异化点 | 说明 | 竞品现状 |
|---|---------|------|---------|
| 1 | AI 深度集成 | 区别于 Habitica 的纯手动记录，引入 AI 目标分解和难度自适应 | 仅有 PlotPlayer 有类似能力 |
| 2 | 柳比歇夫精神 | 强化"时间统计 → 自我认知 → 持续改进"的闭环，而非单纯任务管理 | 无直接竞品 |
| 3 | 精力管理核心 | 填补开源项目空白，实现精力峰值与任务难度自动匹配 | **无成熟方案** |
| 4 | 治未病时间管理 | 预防任务拖延恶化至紧急区间 | **无成熟方案** |
| 5 | 多源数据输入 | 结合可穿戴设备 API 自动更新属性和精力面板 | BeaverHabits 有基础支持 |
| 6 | 降级机制 | AI 不可用时本地模板兜底 | 多数竞品无此设计 |
| 7 | 自托管优先 | 隐私保护和数据主权，区别于 SaaS 方案 | BeaverHabits 有类似架构 |
| 8 | 拖延症自救模式 | 针对拖延症患者的激进干预策略 | **无成熟方案** |

## 二、核心需求与功能规格

### Requirement 1: RPG 式习惯追踪系统
**优先级**：★★★★★ | **参考项目**：Habitica（14k Stars，12 年成熟项目）

系统 SHALL 提供角色化体验，用户的行为转化为游戏中的经验值（XP）和金币，支持等级提升、装备收集、宠物孵化等 RPG 元素。

| 场景 | 触发条件 | 系统响应 |
|------|---------|---------|
| 完成日常任务获得 XP | 用户标记待办任务为完成 | 根据任务难度自动分配 XP 并累积到经验条 |
| 等级提升解锁新内容 | 用户累计 XP 达到等级阈值 | 提示升级并解锁新功能（购买装备、加入组队副本） |
| 失败惩罚机制 | 用户连续跳过日任务 | 扣除 HP，降低等级上限，模拟"共同伤害"机制 |

**任务类型体系**：

| 类型 | 说明 | 游戏化映射 |
|------|------|-----------|
| Daily | 日常任务 | 每日修炼 |
| Habit | 习惯追踪 | 持续累积属性加成 |
| Quest | 一次性任务 | 支线任务 |
| Boss | 阶段目标 | Boss 战 |

**精力标签系统**（新增）：

| 标签 | 说明 | 匹配场景 |
|------|------|---------|
| 创造性任务 | 需要创意产出的工作 | 创造力峰值时执行 |
| 分析性任务 | 需要逻辑推理的工作 | 认知负荷适中时执行 |
| 机械性任务 | 重复性低认知工作 | 精力低谷时执行 |
| 社交性任务 | 需要人际互动的工作 | 社交能量充足时执行 |

---

### Requirement 2: AI 驱动的目标分解引擎
**优先级**：★★★★★ | **参考项目**：PlotPlayer（专注羊）- AI 自动拆解 + 双轨奖励

系统 SHALL 集成 AI 智能体（GoalDecomposer），将用户输入的中长期目标自动拆解为 6-8 个递进式里程碑，每个里程碑进一步拆分为 10/25/40 分钟梯度子任务。支持难度自适应和降级机制。

| 场景 | 触发条件 | 系统响应 |
|------|---------|---------|
| AI 拆解大目标 | 用户输入"我要在三个月内学会 Python 数据分析" | 生成里程碑链（环境搭建→基础语法→数据处理→可视化），并为每个里程碑分配时间梯度任务 |
| 目标难度自适应 | 用户连续完成或跳过任务 | AI 动态调整后续任务的难度和时长推荐 |
| AI 不可用降级 | AI 服务不可用或网络异常 | 使用本地模板兜底，确保核心功能可用 |

### Requirement 3: 生活属性与精力可视化仪表盘
**优先级**：★★★★★ | **参考项目**：地球 Online 看板（15 种属性/实时波动/颜色编码）

系统 SHALL 提供分类生活属性追踪面板，支持三大类属性，**新增精力管理维度**。

**三大类属性体系**：

| 类别 | 包含属性 | 示例 |
|------|---------|------|
| 生理需求 | 睡眠、饮食、运动 | 睡眠时长、饮水量、步数 |
| 情感状态 | 情绪、压力、社交 | 快乐值、焦虑指数、社交能量 |
| 能力属性 | 技能、知识、创造力 | 编程能力、阅读积累、创意指数 |

**精力管理维度**（新增）：

| 维度 | 说明 | 计算方式 |
|------|------|---------|
| 创造力指数 | 评估特定时间段内的创意产出能力 | 基于任务类型与完成质量 |
| 认知负荷 | 衡量大脑处理复杂信息的负荷程度 | 基于专注时长与中断频率 |
| 疲劳程度 | 追踪精力衰减曲线 | 基于活动类型、睡眠数据、连续工作时长 |

**睡眠与恢复维度**（新增）：

| 维度 | 说明 | 数据源 |
|------|------|-------|
| 睡眠质量 | 深睡/浅睡/REM 周期 | 可穿戴设备 API |
| 恢复效率 | 评估休息/午睡/运动对精力的恢复效果 | 系统计算 |
| 最佳休息时机 | AI 建议下一轮专注前的最佳休息类型和时长 | AI 分析 |

**每日启动精力评估流程**：

```
用户启动应用
    ↓
读取：睡眠数据 + 起床时长 + 昨日专注总时长
    ↓
计算：创造力指数 + 认知负荷 + 疲劳程度
    ↓
输出：今日最佳任务类型建议
    （创造力峰值 → 写作/创意）
    （低精力 → 整理/回复邮件）
```

### Requirement 4: 治未病时间管理（拖延预防系统）
**优先级**：★★★★☆ | **参考项目**：SuperProductivity（20k Stars）+ LiFE RPG AI 建议

系统 SHALL 实现"治未病"理念，通过 AI 主动识别重要但未紧急的任务，在任务落入"紧急且重要"区间之前发出预警和干预建议。

**游戏化映射**（艾森豪威尔矩阵）：

| 四象限 | 游戏化命名 | 奖励机制 |
|-------|-----------|---------|
| 重要不紧急 | 修炼任务 | 完成积累长期属性加成 |
| 重要且紧急 | Boss 战 | 高难度、高奖励、触发紧急事件 |
| 紧急不重要 | 杂役任务 | 低 XP 奖励、快速完成 |
| 不重要不紧急 | 闲逛时间 | 消耗金币兑换奖励 |

| 场景 | 触发条件 | 系统响应 |
|------|---------|---------|
| 重要事务预警 | 重要任务距离截止日期 < 3 天且未启动 | 弹出"修炼危机"预警，提升为 Boss 战，建议拆解 |
| 拖延行为识别 | 用户连续推迟某任务 > 2 次 | 触发"治愈建议"，分析拖延原因，推荐调整方案 |
| 主动干预提醒 | 今日任务多为"紧急不重要"类 | 提醒"用战术的勤奋掩盖战略的懒惰" |
| 拖延症自救模式 | 用户手动启用 | 激进干预：30 分钟检查 + 5 分钟步骤拆解 + 自动降级 |

### Requirement 5: 全维度成长可视化
**优先级**：★★★★☆ | **参考项目**：PlotPlayer 成长模块 + Habitica 时间线

| 模块 | 说明 |
|------|------|
| 学习时间线 | 按日期展示每日任务完成情况、专注时长、剧情解锁记录 |
| 专注热力图 | 连续专注天数可视化 |
| 成就徽章墙 | 15+ 成就类型，解锁动画 |
| 等级经验曲线 | 历史等级变化趋势 |

### Requirement 6: 认知负荷与任务匹配系统
**优先级**：★★★★☆ | **参考项目**：无成熟开源方案（**重大空白**）

系统 SHALL 基于用户的精力状态，动态推荐适合执行的任务类型和难度。

| 场景 | 触发条件 | 系统响应 |
|------|---------|---------|
| 每日启动精力评估 | 用户每天启动应用 | 显示"今日最佳任务类型"建议 |
| 认知负荷过高中断提醒 | 连续专注 > 90 分钟且中断次数增加 | 提示"认知负荷已达峰值"，推荐休息类型 |
| 精力低谷期任务降级 | 用户精力指数低于阈值 | AI 自动拆解为简单子步骤，或推荐替代任务 |

**休息推荐逻辑**：

| 负荷状态 | 推荐休息类型 | 预计恢复时间 |
|---------|------------|------------|
| 轻度疲劳 | 5 分钟拉伸 | 5 分钟 |
| 中度疲劳 | 15 分钟冥想 | 15 分钟 |
| 重度疲劳 | 20 分钟午睡 | 20-30 分钟 |

## 六、任务分解与依赖关系

### Phase 1: 核心架构与 RPG 系统

- **任务1: 搭建基础架构**
  - 子任务1.1: 确定技术栈选型
  - 子任务1.2: 设计用户模型（角色属性、等级、经验值、金币、装备、宠物）
  - 子任务1.3: 创建数据库 Schema（用户、任务、习惯、成就、剧情事件）
  - 子任务1.4: 实现基础 API（用户认证、数据 CRUD、进度同步）
  - 子任务1.5: 搭建前端仪表盘骨架

- **任务2: 实现 RPG 式习惯追踪系统**

## 八、AI Agent 框架分析

### 评估背景

本项目需要 AI Agent 框架支撑以下核心能力：
1. **目标分解**：将中长期目标自动拆解为里程碑和子任务
2. **剧情生成**：沉浸式第二人称叙事
3. **难度自适应**：基于用户历史数据动态调整任务
4. **精力评估**：综合分析睡眠、专注、疲劳数据
5. **评估监控**：模型输出质量对比和性能监控

### 候选框架

| 框架 | 定位 | 核心技术栈 |
|------|------|-----------|
| PydanticAI | 轻量级推理框架 + 类型安全验证 | Python |
| CrewAI | 多 Agent 编排框架 | Python |
| Agently | 企业级 Agent 平台 + 内置评估 | Python |
| LlamaIndex | 知识库 RAG 框架 | Python/TS |
| Vercel Eve | 文件系统驱动的 Agent 编排工具 | TypeScript |
| LangChain | 完整 GenAI 生态 | Python/TS |
| LangGraph | 图状态机编排 | Python/TS |

**权重应由项目需求推导**，而非主观分配：

### Eve vs Agently 深度对比

这是 v2 报告与批判性分析的核心争议点。

| 对比维度 | Agently | Vercel Eve | 项目需求匹配 |
|---------|---------|-----------|------------|
| **自托管** | ✅ 完全支持 | ❌ 依赖 Vercel | 需要自托管优先 |
| **确定性流程** | ✅ 进程编排（显式控制） | ❌ LLM 自动路由（不可控） | 需要确定性目标分解流程 |
| **内置评估** | ✅ 框架原生评估系统 | ⚠️ 依赖 Vercel Deploy | 需要开箱即用评估 |
| **内存管理** | ✅ 短期/长期/外部三种策略 | ✅ 持久化执行 | 需要用户成长数据持久化 |
| **学习成本** | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐⭐ 极低 | MVP 阶段可接受学习成本 |
| **国内模型适配** | ✅ 模块化，可插拔 | ⚠️ 需适配层 | 需要适配通义千问/智谱GLM |

**关键判定**：

> Eve 的优势（学习成本低）对本项目不是核心需求，Agently 的优势（自托管、确定性流程、内置评估）才是。
>
> - **自托管冲突**：spec.md 明确要求"自托管优先"，Eve 的 Vercel Deploy 是障碍
> - **不可控流程**：项目需要"确定性流程"（目标分解有确定步骤），Eve 的 LLM 自动路由是风险
> - **适配成本高**：Python 项目使用 TypeScript-only 框架，需编写适配层

**Eve 的价值不在"不用"，而在"借鉴"**：文件系统驱动的设计哲学、人类审批门控的概念、多渠道接入的思想可以应用到 Python 项目中。

### 推理 API 抽象化的澄清

v2 报告的前提"推理 API 已高度抽象化，底层实现可以是外部接口，随时替换"存在偏差。

**实际情况**：
- 通义千问、智谱 GLM、百川的 API 格式各异
- 切换涉及 prompt 适配、token 限制差异、输出格式变化
- 项目需要的是"能切换"，不是"随便切换"
- 关键不在于"能不能换"，而在于"切换成本多高"

**解决方案**：建立统一模型路由层，定义抽象接口，各模型实现独立模块，切换时只需修改配置和少量适配代码。

```python
# 统一模型路由层设计
class ModelConfig(BaseModel):
    provider: Literal['qwen', 'zhipu', 'baichuan', 'moonshot']
    model_name: str
    base_url: str
    api_key: str
    temperature: float = 0.7
    max_tokens: int = 2048

class ModelRouter:
    """统一模型路由 — 通过抽象接口调用外部模型"""
    # 各 provider 实现独立模块
    # 切换时只需修改配置
```
